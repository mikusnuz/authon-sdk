# Authon SDK Audit

Date: 2026-04-06

## Scope

- Primary focus: `packages/js`, `swift`
- Secondary review: `packages/react-native`, `packages/node`, `python`, `go`, `dart`, `kotlin`
- Server-side reference for API parity: `../authon/apps/api`

## Executive Summary

현재 `authon-sdk`에서 가장 신뢰할 수 있는 레퍼런스 구현은 `@authon/js`와 `@authon/node`입니다. Swift SDK는 기본적인 로그인 세션 흐름은 갖추고 있지만, 이메일 인증과 passkey처럼 실제 서비스에서 중요한 분기 처리가 빠져 있습니다. 그 외 일부 서버 SDK는 단순히 기능이 적은 수준을 넘어서, 실제 서버 API와 이미 드리프트가 발생해 런타임에서 깨질 가능성이 높습니다.

가장 먼저 손봐야 할 영역은 다음 다섯 가지입니다.

1. Non-Node 서버 SDK API 경로와 webhook 검증 방식 재동기화
2. 브라우저 JS 세션 refresh 실패 시 auth state 동기화
3. Swift 이메일 인증 플로우 추가
4. Swift passkey 기능의 실제 구현 또는 문서에서 기능 제거
5. SDK별 storage namespace 강화

## Key Findings

### [P1] Swift passkey API is placeholder-only

- Evidence: `swift/Sources/Authon/Authon.swift:266-314`
- Problem:
  `registerPasskey`와 `authenticateWithPasskey`가 아직 TODO 상태이며, 실제 플랫폼 credential 생성 없이 서버가 내려준 options를 그대로 verify endpoint로 다시 보내고 있습니다.
- Why it matters:
  README에서는 passkey를 지원하는 것처럼 보이지만, 현재 구현으로는 실사용 WebAuthn 플로우가 성립하지 않습니다.
- Impact:
  Swift 앱에서 passkey 기능을 켜면 동작 보장이 없고, 문서와 실제 SDK 동작이 어긋납니다.

### [P1] Swift sign-in/up cannot handle verification-required responses

- Evidence: `swift/Sources/Authon/Authon.swift:116-143`
- Related reference: `packages/js/src/authon.ts`
- Problem:
  Swift `signIn`/`signUp`은 항상 토큰 페이로드가 돌아온다고 가정하고 `ApiAuthResponse`만 디코딩합니다. JS SDK에는 있는 `needsVerification` 분기와 `verifyEmail`/`resendVerificationCode` API가 Swift에는 없습니다.
- Why it matters:
  이메일 인증이 필수인 프로젝트에서는 가입 또는 로그인이 정상 완료되지 않습니다.
- Impact:
  Swift SDK를 실제 앱에 붙였을 때, 인증 정책이 조금만 강화되어도 주요 로그인 플로우가 막힐 수 있습니다.

### [P1] Browser auto-refresh can silently invalidate auth state

- Evidence: `packages/js/src/session.ts:105-163`
- Related wrappers:
  `packages/react/src/AuthonProvider.tsx`
  `packages/vue/src/plugin.ts`
  `packages/svelte/src/store.ts`
  `packages/angular/src/service.ts`
- Problem:
  브라우저 SessionManager는 refresh 401 또는 재시도 소진 시 토큰과 storage를 내부에서 직접 지우지만, 이 경로에서 `Authon` 레벨의 `signedOut` 또는 `tokenRefreshed` 이벤트를 발생시키지 않습니다.
- Why it matters:
  상위 wrapper들은 `signedIn`/`signedOut` 이벤트를 기준으로 상태를 유지하므로, UI는 로그인 상태로 남아 있는데 내부 토큰은 이미 사라진 불일치 상태가 생길 수 있습니다.
- Impact:
  세션 만료, 네트워크 장애, refresh 경쟁 상황에서 사용자 경험과 보안 상태가 함께 흔들립니다.

### [P1] Non-Node server SDKs are out of sync with the real API

- Evidence:
  `python/src/authon/client.py:29-72`
  `go/authon.go`
  `dart/lib/src/client.dart`
  `kotlin/src/main/kotlin/dev/authon/sdk/AuthonBackend.kt`
- Server reference:
  `../authon/apps/api/src/modules/backend-users/backend-users.controller.ts`
  `../authon/apps/api/src/modules/sdk-auth/sdk-auth.controller.ts`
- Problem:
  Python, Go, Dart, Kotlin SDK들이 실제 서버에 없는 `/v1/users` 또는 `/v1/auth/verify` 경로를 호출합니다. 반면 현재 서버와 Node SDK는 `/v1/backend/users`와 `/v1/auth/token/verify`를 기준으로 맞춰져 있습니다.
- Why it matters:
  이 차이는 단순 naming 차이가 아니라, 실제 런타임에서 사용자 관리와 토큰 검증이 실패하는 수준의 불일치입니다.
- Impact:
  Node를 제외한 백엔드 SDK는 "지원 언어"처럼 보여도, 현재 서버 기준으로는 깨질 확률이 높습니다.

### [P2] React Native client assumes every sign-in returns tokens

- Evidence: `packages/react-native/src/client.ts:109-126`
- Problem:
  React Native `signIn`/`signUp`은 JS SDK처럼 `mfaRequired`나 `needsVerification` 같은 중간 상태를 처리하지 않고, 바로 `ApiAuthResponse`로 간주해 토큰을 저장합니다.
- Why it matters:
  MFA 또는 이메일 인증이 필요한 프로젝트에서는 정상적인 다음 단계 대신 잘못된 세션 상태가 만들어질 수 있습니다.
- Impact:
  JS SDK와 비교했을 때 public API와 runtime behavior가 일관되지 않습니다.

### [P2] Swift keychain storage is globally shared across projects

- Evidence: `swift/Sources/Authon/SessionManager.swift:10-19`
- Problem:
  Swift SessionManager는 keychain service/account를 고정 문자열 하나로 사용합니다.
- Why it matters:
  같은 앱 안에서 test/live 전환 또는 여러 Authon project를 사용할 경우 세션이 서로 덮어쓰이거나 잘못 삭제될 수 있습니다.
- Impact:
  storage namespace가 약해서 멀티 환경 앱에서 예상치 못한 로그아웃 또는 세션 충돌이 생길 수 있습니다.

## Category Audit

### 1. Hardcoded Values

하드코딩 자체가 항상 문제는 아니지만, 다음 값들은 운영 리스크와 연결될 가능성이 있습니다.

- JS refresh 관련 상수
  `packages/js/src/session.ts`
  `MAX_REFRESH_RETRIES = 3`
  `RETRY_DELAYS = [3, 10, 30]`
  refresh pre-buffer `60s`
  minimum schedule delay `30s`
- JS Turnstile script URL 고정
  `packages/js/src/authon.ts`
- Swift OAuth callback scheme 고정
  `swift/Sources/Authon/OAuthManager.swift`
  현재 `authon://oauth-callback` 기반
- Swift branding cache TTL 고정
  `swift/Sources/Authon/BrandingManager.swift`
- Swift keychain namespace 고정
  `swift/Sources/Authon/SessionManager.swift`

특히 위험한 항목은 callback scheme와 storage namespace입니다. callback scheme는 앱 간 충돌 가능성이 있고, storage namespace는 멀티 프로젝트 구성을 바로 깨뜨릴 수 있습니다.

### 2. Session Stability

세션 안정성은 "refresh가 되느냐"보다 "실패했을 때 상태가 얼마나 일관되게 정리되느냐"가 더 중요합니다.

- JS는 instance 내부에서만 single-flight를 보장합니다.
  `packages/js/src/session.ts`
  `refreshInFlight`로 같은 인스턴스 내 중복 refresh는 막지만, 탭 간 경합까지는 막지 못합니다.
- JS retry 로직은 이름에 비해 공격적이지 않습니다.
  `3s`, `10s` backoff를 선언해 두었지만 minimum delay `30s` 때문에 실제 체감 동작은 많이 달라집니다.
- refresh 실패 시 상위 auth state가 갱신되지 않습니다.
  이 부분이 현재 가장 큰 세션 안정성 문제입니다.
- React Native는 refresh 실패 시 즉시 session을 지우지만 retry가 약하고, 상위 UI state와 동기화도 충분하지 않습니다.
- Swift는 foreground 진입 시 refresh를 시도하는 구조가 있으나, storage/event 설계가 단순해서 예외 상황 observability가 약합니다.

### 3. UX Completeness

JS modal과 browser SDK가 가장 완성도가 높고, Swift가 그 다음, 나머지는 일부 기능이 빠져 있습니다.

- JS:
  전반적으로 가장 많은 auth 분기를 처리합니다.
- Swift:
  UI는 정돈되어 있지만 이메일 인증 플로우가 비어 있고 passkey는 placeholder입니다.
- React Native:
  기본 로그인은 있으나 MFA/verification 중간 상태 UX가 부족합니다.
- React:
  `Forgot password?` 버튼이 보이지만 실제 동작 연결이 없습니다.
  File: `packages/react/src/components/SignIn.tsx`

즉, 화면은 있어 보여도 "실제 서비스에서 끝까지 흐르는 플로우" 기준으로 보면 JS 외 SDK는 아직 빈 구간이 있습니다.

### 4. SDK Consistency

가장 큰 구조적 문제는 언어별 public API와 서버 프로토콜이 동일하지 않다는 점입니다.

- JS는 `needsVerification`과 MFA를 상대적으로 잘 처리합니다.
- Swift와 React Native는 같은 인증 정책에서 다른 결과를 냅니다.
- Node backend SDK는 서버와 맞지만, Python/Go/Dart/Kotlin은 API route가 어긋나 있습니다.
- Webhook 검증 방식도 Node만 서버와 잘 맞고, 다른 언어 SDK는 `timestamp.body` 서명 규약과 `v1=` prefix 처리에서 drift가 있습니다.
- Nuxt는 `useAuthon()` export가 사실상 throw-only stub입니다.
  File: `packages/nuxt/src/composables.ts`
- JS 이벤트 타입과 실제 emit도 완전히 맞지 않습니다.
  `verificationRequired`는 emit되지만 타입에 없고, `tokenRefreshed`는 타입에 있는데 실제 emit이 보이지 않습니다.

결론적으로 "모든 SDK가 같은 public API와 동작을 제공한다"고 보기 어렵습니다.

### 5. Storage Safety

storage는 정상 동작할 때보다 환경이 복잡해질 때 문제가 드러납니다.

- JS / React Native:
  storage key가 `publishableKey` 일부 문자열에 의존합니다.
  `packages/js/src/session.ts`
  `packages/react-native/src/client.ts`
- Swift:
  keychain service/account가 전역 상수입니다.
  `swift/Sources/Authon/SessionManager.swift`

위 방식은 아래 상황에서 충돌 위험이 있습니다.

- test / live 전환
- 여러 Authon project를 하나의 앱에서 사용
- base URL 변경이 섞이는 개발 환경
- publishable key prefix가 우연히 겹치는 경우

권장 방향은 `baseUrl + project identifier + environment`를 반영한 stable namespace입니다.

## Additional Notes

### Event Surface

이벤트 설계가 SDK마다 고르지 않습니다.

- JS:
  이벤트 타입 정의와 실제 emit이 일부 어긋납니다.
- Swift:
  `AuthonEvent` enum은 여러 상태를 정의하지만, public listener 인터페이스는 `AuthonUser?` 중심이라 payload 설계가 어색합니다.
- React Native:
  이벤트 표면이 좁아서 intermediate auth state를 표현하기 어렵습니다.

### Test Coverage

자동화 검증은 JS와 Node 중심입니다.

- Root E2E는 JS auth flow와 Node backend 사용 사례를 주로 검증합니다.
- Swift, React Native, Python, Go, Dart, Kotlin 전용 end-to-end 검증 루프는 충분히 보이지 않았습니다.

즉, 지금 구조는 "레퍼런스 SDK만 강하게 검증되고, 나머지는 문서상 지원"에 가까운 편입니다.

## Verification Performed

아래 명령은 이번 리뷰 세션에서 직접 실행해 확인했습니다.

- `pnpm build`
- `swift build`
- `go test ./...`
- `python3 -m py_compile python/src/authon/*.py python/src/authon/middleware/*.py`

모두 빌드 또는 문법 검증 기준으로는 통과했습니다. 다만 위 findings의 다수는 컴파일 에러가 아니라 런타임 상태 전이, API mismatch, 또는 UX flow 결손 문제이므로 빌드 통과만으로 안전하다고 볼 수는 없습니다.

실제 OAuth, mobile callback, webhook 서명 검증, multi-SDK parity E2E는 외부 설정과 실제 서버 키가 필요해 이번 세션에서는 실행하지 않았습니다.

## Recommended Fix Order

1. Python/Go/Dart/Kotlin SDK의 API route와 webhook verification을 현재 서버 규약에 맞게 재동기화
2. JS SessionManager refresh 실패 시 `signedOut` 또는 동등한 상태 이벤트를 상위에 전달
3. Swift에 `needsVerification` 대응과 `verifyEmail` / `resendVerificationCode` API 추가
4. Swift passkey를 실제 플랫폼 credential 기반으로 구현하거나, 지원 문구를 문서에서 제거
5. JS, React Native, Swift storage namespace를 프로젝트/환경 단위로 분리

## Bottom Line

현재 `authon-sdk`는 "모든 SDK가 동일한 수준으로 안정적"이라고 보기 어렵습니다. 실전 기준으로는 JS와 Node가 기준선이고, Swift는 빠르게 보강 가능하지만 중요한 auth branch가 비어 있으며, 나머지 일부 SDK는 이미 서버와의 계약 자체가 흔들리고 있습니다.
